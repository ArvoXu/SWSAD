## 資料庫重建
那我先從資料庫遷移開始講，因為後面的高併發和數據一致性問題，其實都是建立在一個好的資料庫架構基礎上。

**舊系統長什麼樣？**
舊的會員系統，用的是Excel式的構建，就是一張表裡面，會員編號、方案、資產這些全部混在一起。但關鍵問題來了，同一個人，可能因為需要購買不同方案，有多個會員編號。比如說，同一個手機號 09-3700-0001，可能在系統裡被記成不同的形式，不同的變體。這就意味著，同一個人的資產全部分散在這些帳號裡面，根本沒有一個完整的視圖。

再加上，上下線關係也只是用「經銷商編號」這個欄位來記，無法清楚地建立起整個分銷網絡的樹狀結構。這對於後面要算加點、分析績效等等全部都是麻煩。

**我怎麼改的？**
我思路是把「人」和「帳號」分開，並且讓訂閱和帳戶綁定。舊系統混在一起，新系統是這樣的：

```
會員層級 (member)
  ├─ 代表一個真實的人
  ├─ 用手機號歸一化，去掉所有特殊字符，只取最後 9 碼
  └─ 用一個邏輯會員 ID（M000001）來代表

帳戶層級 (account)
  ├─ 代表這個人在系統裡的每一個登入帳號
  ├─ 一個會員可能有多個帳號（同人多帳）
  ├─ 但每個帳號的資產各自獨立記錄
  └─ 保留原始的會員編號，這樣可以回溯

訂閱層級 (subscription)
  ├─ 記錄每個帳號購買的方案和時間
  └─ 帳號和訂閱是一對一的

```

**為什麼這樣設計？**

首先，資產零丟失。每個帳號的虛擬點數、積分我都各自保留，所以即便同人多帳，加起來的資產總額和遷移前完全一致，我用驗證腳本確認過。

其次，上線關係有了真實的邏輯。我不是隨便用帳號編號去匹配，而是看「訂閱時間有沒有重疊」加上「經銷商編號對不對」。時間重疊、編號也對，那就標記為 high confidence，只有編號對、時間沒重疊的，才標記為 account_only。這樣上線網絡的邏輯就清楚了。

第三，有了審計線索。舊的會員編號我沒有丟，記在 account 表的 orig_account 欄位，以及 member 表的 notes 裡面寫著是從哪些舊編號合併過來的。將來要回溯問題，或者老闆問「為什麼這個會員的資產數字」，我都能查得到。

**整個遷移的工程邏輯**

我的 migration 目錄裡面，有四個主要的 Python 腳本：

- **convert_multi_account.py** - 這是主引擎，負責手機號歸一化、同人帳號分組、上線推導的核心邏輯
- **validate_accounts.py** - 確保遷移出來的帳號資料完整有效，不會有重複 ID 或者資產為負數這種情況
- **validate_subscriptions.py** - 驗證訂閱紀錄的時間順序、欄位關聯、上下線的邏輯一致性，確保沒有環路或者騎虎難下的情況
- **generate_referral_graph.py** - 把上下線關係轉成 GraphViz 的圖譜，這樣可以視覺化地看整個分銷網

## MLM會員系統中的高併發處理

所以再來講第二個重點，高併發。為什麼會員系統需要考慮高併發呢？

最核心的場景就是**點數轉帳**。在 MLM 系統裡面，會員之間可以互相轉虛擬點數，也可以透過掃 QR Code 完成商戶支付，這些都涉及到帳戶餘額的扣減和增加。想像一下，同時有一百個會員在掃別人的 QR Code 做轉帳，或者說一個人有多個帳號要同時處理轉帳請求，這時候就會產生競爭，如果沒有好好控制，就很容易出現帳戶餘額被重複扣減，或者某個轉帳被處理兩次的狀況。

**我的處理思路**

我的處裡方法是資料庫層級解決方案：**交易加上row level locking**。

具體來說，一筆轉帳請求進來時，系統會先開啟一個資料庫交易，然後立刻對涉及的兩個帳戶—from account和to account—加上 UPDATE 鎖。這個鎖是row level的，不是table level的，所以不會影響其他帳戶的操作。只要有其他請求想要操作同一個帳戶，就必須等待這個鎖被釋放。

鎖定住帳戶之後，系統才開始驗證餘額、計算手續費，然後扣點數、增加數。整個過程是一個不可分割的單位，要嘛全部成功，要嘛全部失敗。如果中途任何一步出錯—比如帳戶不存在、餘額不足—整個轉帳就會回滾，帳戶狀態就像這筆轉帳從來沒有發生過一樣。

**幂等性保障**

還有一個**idempotency**。一筆轉帳如果因為網路等等原因不小心被提交了兩次，系統應該要認出來這是重複的請求，而不是兩筆不同的轉帳。

我的做法是在轉帳開始前就生成一個唯一的 key，這個 key 包含了from account、to account、金額和時間，然後把它存進資料庫。如果後來又有一個相同的請求進來，系統就會查詢資料庫，發現 key 已經存在過了，那就一樣直接返回之前的結果，而不是再執行一次。

**連接池**

在基礎設施層級，我調整了資料庫連接池的參數。連接池的最大連接數設在 100，最小閒置連接數是 10。這樣做的目的是確保有足夠的連接去應對並發請求，但又不會因為連接太多導致資源浪費。設計上，一個請求進來，它會從池裡面借一個連接，執行轉帳邏輯，然后把連接還回池裡。

總結一下，高併發在本專案的核心就是鎖定住有爭議的資源，然後在確保沒有人會在我操作的時候偷偷改東西的前提下，才敢去改帳戶餘額。

## MLM會員系統中保持數據一致性

現在來講第三點，數據一致性。這和高併發其實是互相關連的，但我分開講是因為它涉及更多維度的考慮。

**為什麼是難題？**

假設一個會員從 A 帳號轉 1000 點給 B 帳號。表面上看就是一個轉帳，但實際上牽涉到三個地方都要被改動：

1. A 帳號的點數要減少 1000
2. B 帳號的點數要增加（1000 減去手續費）
3. 系統帳戶的手續費收入要增加

這三個操作要嘛全部成功，要嘛全部失敗。想像一下，如果第一步減少 A 帳號成功了，但第二步增加 B 帳號因為某個錯誤失敗了，那麼點數就從整個系統裡面無故消失了。另一個場景是，A 帳號被凍結了，但系統沒有察覺到，結果仍然扣減了 A 帳號的點數，這樣就造成了賬務混亂。

**我的解決方案**

我用的是資料庫交易來保證**原子性**，也就是 ACID 裡面的 A。

首先，整個轉帳流程被包裹在一個交易裡面。如果三個操作都成功了，交易就提交，資料庫會永久記錄這個改動。如果任何一步失敗了，我就呼叫回滾，所有的改動都會被撤銷，就像交易從來沒有開始過一樣。

其次，在資料庫層級，我還加上了**完整的審計紀錄**。每一筆轉帳，系統都會建立三筆轉帳紀錄—一筆記錄 A 帳號的出帳、一筆記錄 B 帳號的入帳、一筆記錄系統帳戶的費用收入。這樣就算將來某個時刻有人問「為什麼 A 帳號少了 1000 點」，我可以直接查詢轉帳紀錄，告訴他具體是什麼交易導致的。

**資料結構的支持**

我在設計資料庫時，是這樣分層的：

```
帳戶層級 (Account)
  ├─ 儲存當前的點數餘額（MP Balance、VP Balance）
  ├─ 一個帳戶對應一筆訂閱

轉帳紀錄層級 (PointTransaction)
  ├─ 記錄每一筆點數的增減事件
  ├─ 包含轉帳類型（出帳、入帳、手續費等）
  ├─ 有唯一的 transaction_key 確保不重複記錄

轉帳請求層級 (P2PTransfer、MPTransferRequest)
  ├─ 記錄轉帳的狀態（進行中、已完成、失敗）
  ├─ 用 transaction_key 去重，防止同一筆轉帳被處理兩次
```

**與高併發的配合**

當高併發和一致性功能結合時，就變成了這樣的流程：

1. 請求進來時，系統先生成唯一的交易 key
2. 開啟資料庫交易
3. 鎖定轉帳雙方的帳戶行
4. 驗證帳戶是否存在、餘額是否足夠
5. 查詢是否有重複的交易 key（幂等性檢查）
6. 執行轉帳：扣減 A、增加 B、記錄三筆轉帳紀錄
7. 送出系統帳戶的手續費
8. 建立轉帳紀錄並標記為完成
9. 提交交易

整個流程裡，鎖定、驗證、幂等檢查、余額扣減、記錄紀錄這些全部都在同一個交易內部完成。如果某一步失敗了，整筆交易回滾，這樣就確保了帳目永遠不會出現不平衡的狀態。

## SWSAD系統 - 「三點之間」模型

最後，我想分享一下我對儀表板設計的理解。其實多數 dashboard 都不是真正的決策系統，而只是資訊展示系統。好的儀表板應該要做什麼呢？我把它簡化成一個模型叫「三點之間」。

**三點之間模型**

```
使用者 -> 資訊 -> 決策
```

中間有兩段距離：

第一段是「使用者到資訊」的距離，我叫它 Accessibility。你得花多少步、花多少時間，才能拿到資料。如果一個 dashboard 要點七八層才找到你想要的東西，Accessibility 就爛透了。

第二段是「資訊到決策」的距離，我叫它 Interpretability。拿到資料後，你要花多少秒才能把它轉成判斷？如果資料在那邊，但你還是得自己整理、自己對比、自己翻譯，Interpretability 就不夠。

**SWSAD 為什麼這樣設計**

我在做 SWSAD—這是針對我們餐飲設備代理商業務設計的倉儲銷售儀表板—的時候，就是圍繞著縮短這兩個距離。

首先，我觀察了使用者。老闆要看銷售報表，行銷要看活動成效，補貨人員要看即時庫存，行政要能對帳，營運（也就是我）要看流程阻塞點。這些角色各有各的想法，有些甚至是互相衝突的。但如果你仔細看下去，他們真正關心的東西其實只有三件—庫存、銷售、補貨。所以我就沒必要做十個頁面。

其次，我用時間做架構的主軸。使用者其實不是「用功能」在思考問題，而是「用時間」在思考問題：現在發生了什麼、過去發生了什麼、接下來該做什麼。所以我把整個系統拆成三個頁面，分別對應三個決策時點。

**三頁面架構**

- **當下頁**：看此刻的機台庫存，以及最近一個月的產品排名。補貨人員用這頁，幾秒內就能判斷哪台機器缺什麼
- **過去頁**：回看截至今日以前的所有銷售資料。支援多維度交叉比較—時段、機台、產品隨便組合—這樣行政和行銷不用再把資料搬來搬去
- **未來頁**：根據銷售表現提供算法補貨建議。系統會根據設定的策略—穩健平衡、積極增長、探索汰換—自動計算每個產品的補貨數量，整理成補貨單直接用

**具體改善**

最直觀的成果就是時間。我們量過實際操作：

- 行政製作單月報表，原來要 20 分鐘，現在不到 30 秒
- 補貨確認單台機台庫存，原來要 8 分鐘，現在不到 10 秒
- 行銷交叉對比品項表現，原來要 15 分鐘，現在不到 30 秒

這不只是快了幾倍，而是改變了工作流程本身。原本需要切換頁面、記筆記、來回對照的「作業感」，變成了「所見即所得」。單純把頁面堆上去，做不出這種差別。

所以，好的儀表板不是資訊的牆，而是決策的導航。它不該要求使用者去適應系統的分類邏輯，而應該被設計成直接對應使用者的工作流程和決策節奏。這就是我對 dashboard 的理解。
