# 三點之間：用「決策距離」重新設計 Dashboard

多數 dashboard 並不是決策系統，而只是資訊展示系統。在我看來，好的儀表板可以被極致簡化成一個模型：**三點之間**。

```text
使用者 -> 資訊 -> 決策
```

- 使用者到資訊的距離，是 Accessibility：幾步能拿到資料，也就是 `click depth / time-to-data`
- 資訊到決策的距離，是 Interpretability：幾秒能把資料轉成判斷，也就是 `time-to-decision`

如果一個 dashboard 讓人要點很多層才找到資料，Accessibility 太差；如果資料拿到了還要再自己整理、翻譯、比對，Interpretability 就不夠。所有 dashboard 的優化，本質上都應當在此基礎降低這兩種成本。

多數機台廠商都有自己的 dashboard，但實際用起來都像是為了接住所有設備輸出而把東西堆在同一個畫面上，並沒有真正完成幫人判斷的系統職責：資訊很多，卻不好用；功能很多，卻不靈活；畫面很滿，卻不清楚。

這類 dashboard 常見的問題其實很一致：

- 多層 navbar，想看一台機器的庫存，得先點完67個按鈕
- 可視化程度低，資料在，但重點不在
- 警報頻發卻缺乏判讀價值，如製冷設備除霜時跳高溫警報，畫面很緊張，設備卻無事發生
- 資料無法有效篩選，也沒有被整理成能直接使用的形式
- 外觀不友善，操作時帶有強烈「作業感」

最後一點看似不重要，但工作已經是工作了，如果 dashboard 還把壓力再做成 UI 端上來，效率通常更差。反過來說，介面若足夠清楚、輕盈，甚至帶一點親和可愛，使用者的判讀速度、接受度、黏性往往都會更好(比如我在中秋節會發布有月亮和兔子且偏暖色的版本)。

作為餐飲設備代理商，我看過不少廠家的 dashboard。此模型聽起來雖然簡單，但現實是，多數系統通常只解決了一半的問題：要嘛資訊很好拿，但難以解讀；要嘛資料解釋得還行，但取得過程冗長。更常見的是，兩段距離都不短。上述看似分散的抱怨，都是同一件事：**使用者到資訊太遠，資訊到決策也太遠。**

## 把模型落地：SWSAD 的設計思路

**SWSAD（智慧倉儲與銷售分析儀表板）** 是我為餐飲設備代理商開發的營運工具。它的核心目標不是「把資料放上去」，而是縮短不同角色取得關鍵資訊的時間，讓資料真正成為決策捷徑，而非另類負擔。

### 第一步：貼近實務，從角色看需求，也看衝突

從使用者角色出發，觀察這套系統是誰在用、誰要看什麼。

- 老闆
- 行銷
- 補貨
- 行政
- 營運（我）

這些角色雖然都會碰到同一套系統，但關注點完全不同：

- 老闆：是銷售報表與營收成長
- 行銷：短期活動成效與產品表現
- 補貨：即時庫存與補貨優先順序
- 行政：對帳資料與補貨單
- 營運：流程阻塞點與整體效率推進機會

這不只是不同角色的需求列表而已，裡面其實有明顯的 trade-off：

- 行銷希望多測新品與活動成效，但補貨希望降低 SKU 複雜度
- 行政需要資料口徑穩定一致，但營運需要快速調整策略
- 老闆在意營收成長，但現場要承擔執行與補貨成本

再進一步做額外觀察與溝通後，還有幾個關鍵：

- 多數使用者並非技術背景
- 使用者在原系統裡，往往無法快速找到自己需要的資訊
- 銷售與庫存資料常常得靠行政層層點選 navbar，再整理成 Excel 才能用

這代表問題不只是「資料有沒有」，而是「資料能不能在對的時間，被對的人，以合理的成本拿到」。所以我沒有把需求往更多功能頁面拆，而是先把衝突吸收，再把需求壓縮成更少但更準確的決策入口。

### 第二步：壓縮需求成為更明確的架構

分析完角色與需求後，事情就明瞭了。即便不同職位的工作內容不一致，但核心需求高度重疊，最後都可以歸納到三件事：

- 庫存
- 銷售
- 補貨

既然如此，就沒有必要把系統做成層層展開的navbar迷宮。對於多數非技術使用者來說，操作直觀比功能堆疊更重要；而對補貨這類移動場景來說，RWD 也不是加分項，而是基本條件。畢竟不可能有提著筆電在倉庫和機台之間跑的場景。

### 第三步：以時間做量尺

多數 dashboard 習慣照功能做分類，但使用者實際上不是用「功能」在思考問題，而是用「時間」在思考問題：現在發生了什麼、過去發生了什麼、接下來該做什麼。而當一個系統需要使用者先思考「功能在哪」，而不是「現在要解決什麼」時，設計就已然偏離決策流程。

捨棄 navbar 後，我把整個 SWSAD 拆成三個頁面，分別對應三個時間點：

- 當下：查看此刻的機台庫存，以及近一個月的產品排名
- 過去：回看截至今日以前的所有銷售資料
- 未來：根據銷售表現提供算法補貨建議，並加入庫存預警

這樣的拆法不是單純的資訊架構選擇，而是把原本分散在不同角色身上的衝突，收斂成三種最常見的決策場景。使用者不必先理解系統分類，再去找資料；而是可以直接進入自己當下要處理的問題。

### 數據化成果：時間成本的物理壓縮

僅是設計理念，難免顯得主觀，但對一套實際落地的營運系統來說，時間被切了多少才最具說服力。

以日常關鍵操作為例，根據實測，SWSAD 帶來的改善十分具體：

| 任務內容 | 原系統耗時 | SWSAD 耗時 | 效率提升 |
| --- | --- | --- | --- |
| 行政：整理、製作單月分店與產品報表 | ~20 分鐘 | <30 秒 | ~97.5% |
| 補貨：確認單台機台庫存與補貨標的 | ~8 分鐘 | <10 秒 | ~97.9% |
| 行銷：交叉對比特定條件品項表現 | ~15 分鐘 | <30秒 | ~96.7% |
| 營運：產出單週補貨數量與策略建議 | ~10 分鐘 |  <10秒 | ~98.3% |

如果對照前面的模型，這些改善其實就是兩件事同時發生：`time-to-data` 被大幅壓縮，`time-to-decision` 也跟著縮短。這不是單純快了幾倍的差別，而是本質上的流程轉換：原本需要切換頁面、記錄資料、來回對照的作業感，被壓縮成「所見即所得」。對營運來說，這才算真正把 dashboard 從圖表看板轉變成決策導航引擎。

## 具體實作

### 庫存

廢除傳統表格導向的庫存資訊，導入百分比圖示化介面。不同顏色映射不同產品，灰色直觀反映庫存餘量。強制聚焦過去一個月數據，並提供精確排名與數量參考以避免資訊過載，好處很直接：不需要先讀表，再做腦內轉換，使用者一眼就能判斷哪台機器缺什麼、缺多少。

### 銷售

銷售頁面支援多種實用分析維度，包括：

- 各分店銷售額
- 產品銷售占比
- 支付方式占比

同時也支援多維度交叉比較，例如：

- A 日期 + B 機台 + C 產品
- C 日期 + A 機台 + B 產品

這類組合已足以涵蓋多數日常分析需求。此外，系統也能產出月報表，方便行政與會計對帳，不必再把資料搬來搬去重新整理。

### 補貨

補貨模組是我特別重視的一塊，因為它直接連到實際營運成本。這部分我設計了三種策略，並於後端（`server.py`）以具體的權重分配邏輯落地實作：

- **穩健平衡 (`stable`)**：完全按比例分配。根據單品在總銷量中的占比，等比例瓜分可用機台格位，且受限於實際倉庫庫存。
- **積極增長 (`aggressive`)**：資源集中化。強制將 80% 的空間資源全數投入「銷量前三名」的產品，其餘品項僅能平分剩下的 20%。
- **探索汰換 (`exploratory`)**：強迫測試市場。強行保留 20% 的空間分配給倉庫內「沒有銷售紀錄」的新品進行測試，舊品僅能按銷量比例分配剩餘的 80%。

```python
# 後端資源分配核心架構示範
if strategy == 'stable':
    # 依銷量佔比等比分配
    suggested_qty = (sales / total_sales) * available_slots
    
elif strategy == 'aggressive':
    # TOP 3 產品拿走 80% 機台空間
    slots_for_top_3 = round(available_slots * 0.8)
    # 其餘產品均分 20%
    slots_for_others = available_slots - slots_for_top_3
    
elif strategy == 'exploratory':
    # 既有品項分配 80% 空間
    slots_for_existing = round(available_slots * 0.8)
    # 強制保留 20% 資源給無銷售紀錄之新品
    slots_for_new = available_slots - slots_for_existing
```

系統會根據產品銷售狀況與當下倉庫庫存數量，結合上述算法計算補貨方向，降低行銷與補貨人員憑直覺、印象或慣例做決策的比例。而為降低現場作業複雜度，系統將在最後進行收斂，單次建議最多提出 7 項最具效益的變動。這不只是節省時間，也是在降低主觀判斷帶來的偏誤，讓決策空間被限制在有效的範圍內。同時，結果可以直接轉成正式補貨單，銜接後續物流與行政流程。

## 關於進一步擴展

曾嘗試過把ML納入這套系統，將氣溫、降雨、季節、地理位置與地點類型等特徵放進模型，進一步強化補貨與預警判斷。

不過，在當時的資料量條件下，模型在時序交叉驗證中的準確度僅 73.58%，不足以證明它能穩定優於現行的 rule-based 補貨邏輯，更不足以承擔正式營運決策的風險，因此沒有上線。對我而言，模型存在的意義不是讓系統聽起來更厲害，而是要在可驗證的前提下比既有方法更可靠；做不到，就停在實驗階段。會做 ML 不是重點，知道什麼時候不該上 ML，才是系統設計的一部分。

## 結語

好的 dashboard 不是資訊的展示牆，而是決策的導航。它不該要求使用者適應系統，而應該先理解人是怎麼工作、怎麼判斷、怎麼承擔成本，再把資訊整理成適合那個節奏的形式。

SWSAD 對我來說，並不只是把庫存、銷售和補貨放進同一個專案裡，而是一次更明確的系統設計嘗試：讓行政少做報表整理，讓補貨少做來回翻查，讓行銷少做資料搬運，也讓營運少靠印象決策。它的重點不是把更多資訊做上牆，而是把複雜的營運問題，壓縮成更短的理解路徑與更低的決策成本。

當這件事被持續做到極致時，dashboard 的角色就不再是輔助工具，而是實際參與決策的一部分。
